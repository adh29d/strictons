Phase-organised running record of what landed, what we decided, and what surprised us. Maintained at the end of each phase per the convention in `CLAUDE.md`. Append a new section when a phase merges to `main` and is verified live; do not retroactively edit older entries.

## Phase 1 — Monorepo foundation (merged 2026-05-04)

### What landed

- pnpm 10 workspace with Turborepo task pipeline. Node 22 LTS pinned via `.nvmrc` + `packageManager` field in root `package.json`.
- Strict TypeScript everywhere (`noUncheckedIndexedAccess`, `noImplicitOverride`, `no-explicit-any` enforced via lint).
- Three shared config packages under `packages/config/`: `@strictons/config-eslint` (flat config base + Next.js preset, `no-explicit-any` rule), `@strictons/config-tsconfig` (base / nextjs / react-library), `@strictons/config-tailwind` (Tailwind v4 design tokens via CSS-first `theme.css`).
- Two placeholder shared packages: `@strictons/ui` (one `AppShell` component used by every app to verify cross-package imports), `@strictons/types` (placeholder for shared TS types).
- Four Next.js 15 App Router apps on staggered ports so `pnpm dev` doesn't collide: `marketing` (3000), `admin` (3001), `partners` (3002), `mystay` (3003). Each renders the placeholder `AppShell`.
- GitHub Actions CI: `pnpm format:check`, then `pnpm turbo run typecheck lint build`.
- Repo root README with the Vercel four-projects-from-one-repo runbook (separate Vercel project per app, root directory `apps/<name>`, build via Turborepo `--filter` syntax, env-var split between Vercel team-level shared secrets and per-project app-specific values).

### Locked decisions

- **Each app/package owns its own `eslint.config.mjs` consuming the shared preset.** No monolithic root config that knows about every workspace; the root config lints repo-level files only.
- **`packages/config` is a namespace dir, not a single workspace package.** Three separate packages live underneath (`config-eslint`, `config-tsconfig`, `config-tailwind`) because their dependency graphs differ — only `config-eslint` pulls in plugins. `pnpm-workspace.yaml` includes `packages/config/*` so the nested layout works.
- **App ports staggered 3000-3003** so `pnpm dev` running all four apps in parallel doesn't collide.
- **Vercel topology: four separate Vercel projects, one per app, all from this repo.** Lets us attach hotel custom domains only to the `mystay` project, deploy apps independently, and keep env-var blast radius small. The runbook is in `README.md`.
- **Tailwind v4 with CSS-first config.** Shared design tokens live in a single `@theme` block in `packages/config/tailwind/theme.css`; apps `@import` it from `globals.css` after `@import "tailwindcss"`. No `tailwind.config.{js,ts}` files in apps.
- **Sentry SDK NOT wired in Phase 1.** Env-var slot documented in each `.env.example`, real wiring deferred until there are surfaces to instrument.

### Gotchas

- **`next-env.d.ts` is auto-generated** with a triple-slash reference that fails `@typescript-eslint/triple-slash-reference`. Fix: gitignore the file + add `**/next-env.d.ts` to the shared next eslint preset's `ignores`. Discovered when the marketing app's lint started failing after the second app was added.
- **`sharp` postinstall is ignored by pnpm 10** (default-deny on build scripts). Native bindings would speed up Next image optimisation but aren't required. Deferred — added to root `package.json` `pnpm.onlyBuiltDependencies` as a known-future safelist candidate, not enabled yet.
- **`next build` emits "The Next.js plugin was not detected in your ESLint configuration"** — cosmetic warning from `next build`'s legacy-config sniffer. The plugin IS loaded via flat config; the warning doesn't reflect actual lint behaviour. Safe to ignore until Next's CLI catches up to flat config.

### What's deferred

- Sentry SDK instrumentation (env slot present, real wiring later).
- Custom hotel domain attachment to the `mystay` Vercel project (runbook in repo root README; not exercised until a hotel asks).

## Phase 2 — Database & RLS skeleton (merged 2026-05-05)

### What landed

- Thirteen SQL migrations under `packages/db/supabase/migrations/`: `baseline` (extensions, 19 enums with the hotel-approval state-machine doc as a comment block, helper functions), eight cluster migrations covering every table from the brief (users + strictons_staff, hotels cluster, businesses cluster, ad_placements cluster, candidate_businesses, briefs cluster, qr_codes + events + rollup skeletons, audit_log), and four fix-forward migrations (function-backed `social_handles` CHECK, anon write revoke, authenticated write revoke, view write revoke).
- `@strictons/db` package: `createServiceRoleClient()` with explicit "server-side only" header comment, generated `database.types.ts` (1461 lines, scoped to `public` schema), Supabase CLI pinned as a dev dependency.
- pgTAP harness (`_setup.sql` + `_helpers.sql` with role-impersonation and fixture seeders) plus 10 spec files: unauth + structural audit, Strictons staff, hotel and business isolation, audit append-only, quality clause, cross-tenant writes, service-role reads, print-state immutability, premium-position uniqueness.
- Three CI workflows: `db-test.yml` (boot local Supabase → regenerate types → drift check → pgTAP suites; ordered so types generation runs before pgTAP installs to avoid extension pollution), `db-deploy-dev.yml` (auto-apply on push to main), `db-deploy-prod.yml` (manual `workflow_dispatch` only, two-stage diff→push gated by GitHub Environment with required reviewers; inert until prod is provisioned).
- Reference data (mood_options) ships as a migration in a Phase 2 follow-up PR (#4) so `db push` against hosted environments picks it up. Auth-linked dev fixtures stay in `scripts/seed.ts` (local only).
- `packages/db/README.md` documents the layout, migration-authoring conventions, type regeneration, service-role client usage, env-var matrix (with the November 2025 Supabase API key naming: `SUPABASE_SECRET_KEY` / `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`), and the prod provisioning runbook.

### Locked decisions

- **All Supabase access goes through `@strictons/db`.** Apps never import `@supabase/supabase-js` directly. The package is the only seam; helpers and clients live there.
- **Service-role client (`createServiceRoleClient()`) is server-side only.** Multi-line warning comment on the function forbids importing from any `'use client'` module — doing so would inline `SUPABASE_SECRET_KEY` into the browser bundle. Allowed callers: Server Components, Route Handlers, Server Actions, background scripts.
- **Migrations are append-only once they have successfully applied to any shared environment** (green CI, dev, or prod). Edit-in-place is permitted only on a branch where CI has not yet gone green for that migration — there is no environment to preserve. Documented in `packages/db/README.md` with rationale.
- **Reference data ships as a migration with `INSERT ... ON CONFLICT DO NOTHING`, not in `seed.sql`.** `supabase db push` does not apply `seed.sql`; only `supabase db reset` does. Anything every environment needs goes in a migration. Auth-linked dev fixtures (test users, sample hotels) stay in `scripts/seed.ts` and only target local.
- **Views need explicit `revoke insert, update, delete from anon, authenticated`.** The `revoke … on all tables in schema public` form does not reliably extend to views in Supabase's Postgres. Documented in the migration-authoring checklist.
- **`alter default privileges in schema public revoke …` is set for both `anon` and `authenticated`** so future public tables inherit no-write defaults. Migrations that need an authenticated write path on a new table must add an explicit GRANT.
- **Strictons writes route through the service-role client.** No `FOR ALL to authenticated using is_strictons_staff()` policies on Strictons-only tables (`guides`, `print_change_requests`, `candidate_businesses` — the three FOR ALL bypasses dropped during the security audit). Strictons admin-side mutations use the service-role client server-side. Phase 4+ admin app code is consistent with this.
- **pgTAP types must be excluded from generated types via CI ordering.** `gen:types` runs *before* the test runner installs pgTAP into `public`. The schema flag (`--schema public`) does not filter extension-owned objects in our pg-meta version, so ordering is the actual fix.
- **The structural audit query in `tests/01_unauth.spec.sql` (test 21) is the canonical backstop for orphan GRANTs going forward.** Joins `information_schema.role_table_grants` against `pg_policies`, excludes extension-owned objects via `pg_depend`. Counts (table, role, privilege) tuples that have a write GRANT to anon or authenticated with no backing RLS policy permitting that role + operation. Catches new tables added in later phases automatically.

### Gotchas

- **SQLSTATE 0A000 — "cannot use subquery in check constraint."** Postgres rejects any subquery (including SELECT … FROM jsonb_each(…)) inside a CHECK. Wrap the predicate in an `IMMUTABLE` SQL function and reference the function from the CHECK. Bit us on `businesses.social_handles` validation; fix-forward migration `20260504100900_add_social_handles_check.sql` introduced `is_valid_social_handles(jsonb)`.
- **Silent zero rows on UPDATE / DELETE under RLS.** When no policy matches the calling role, INSERT raises (WITH CHECK is mandatory) but UPDATE / DELETE return zero rows with no error. Combined with Supabase's default GRANT-ALL machinery, anon and authenticated could issue write statements that succeed silently against tables they have no policy on. Fix: pair RLS with explicit `revoke insert, update, delete from anon` (blanket-safe) and per-table `revoke` from `authenticated` where no policy permits, plus `alter default privileges` for future tables. The pattern is now codified in the structural audit (test 21).
- **Hosted Supabase direct connections are IPv6-only; GitHub Actions runners have no IPv6.** Migrations from CI must use the **session pooler** connection string for both `SUPABASE_DB_URL_DEV` and `SUPABASE_DB_URL_PROD`, not the "direct connection" string from Settings → Database. The session pooler is IPv4-reachable and accepts the same migration commands.
- **`supabase db push` does not apply `seed.sql`; only `supabase db reset` does.** Reference data shipped only in `seed.sql` will be missing from dev / prod. Reference data must ship as a migration (see locked decisions).
- **pgTAP installs auxiliary objects (`tap_funky`, `pg_all_foreign_keys`) into the `public` schema.** Without filtering, these leak into the generated types AND show up as orphan GRANTs in the structural audit. Fix: in audits, exclude extension-owned objects via `pg_depend` (`classid = 'pg_class'::regclass AND refclassid = 'pg_extension'::regclass AND deptype = 'e'`); for type generation, run `gen:types` before pgTAP is installed in CI.
- **Temp tables across role switches need `grant select, insert on _t to public`.** Postgres's default temp-table ACL grants access only to the creating role. pgTAP suites that `_test_as_user(...)` or `_test_as_service()` and then read a temp table fail with `permission denied for table _t` without the explicit grant. Cleaner long-term fix: avoid temp tables in role-switching suites entirely (suite 09 already does, using a `DO` block + PL/pgSQL variables).
- **`auth.users` vs `public.users` ordering.** Test fixtures must seed `auth.users` first to satisfy the FK from `public.users.id`. The `on_auth_user_created` trigger (`SECURITY DEFINER`) handles the `public.users` insert automatically — seeders should only insert into `auth.users` and never double-insert.
- **`information_schema.role_table_grants` reports a row per table when ANY column has the privilege, including views and column-level grants.** When auditing, expect column-restricted UPDATEs (e.g. `hotels.contact_email`) to appear at the table level — back-check via `pg_policies` to confirm a policy permits the role, not just by counting rows.
- **`pg_policies.cmd` is full-name strings (`'INSERT'`, `'UPDATE'`, `'DELETE'`, `'SELECT'`, `'ALL'`)** in modern Postgres / Supabase, despite older docs implying single-character codes (`r`/`a`/`w`/`d`/`*`). Match against full names in audit queries.
- **Append-only is a rule that earns itself.** A migration that has never reached green CI may be edited on its branch because no environment has been affected by it. We hit this when migration 4 (`businesses.sql`) failed parse-time on a subquery-in-CHECK; the cleanest fix was editing the offending file rather than adding a fix-forward against a table that never got created. The carve-out is now in `packages/db/README.md`.

### What's deferred

- **`strictons-prod` Supabase project provisioning.** Workflow file (`db-deploy-prod.yml`) is in place but inert. Provisioning runbook is in `packages/db/README.md` — create the project, set `SUPABASE_DB_URL_PROD` (session pooler URL) as a repo secret, configure the `production-database` GitHub Environment with required reviewers, dispatch a dry run, then dispatch real apply. Run when Phase 7 or 8 needs it.
- **Custom hotel domains UI.** Architecture is ready (`hotels.custom_domain` field nullable from day one, hostname-or-subpath middleware planned for `mystay`), but self-service UI is deferred per brief §3.1. Manual attachment via the admin runbook covers the first wave of requests.
- **Phase 9 analytics rollups.** Three rollup tables exist as empty skeletons keyed `(hotel_id, guide_id, business_id, event_type, time_bucket)` with `UNIQUE NULLS NOT DISTINCT`. The `pg_cron` jobs that populate them land in Phase 9 alongside the capture endpoints.
- **Cross-cluster `authenticated`-role write audit.** The blanket `anon` revoke is comprehensive; the `authenticated` revoke is per-table. If a future table grants `authenticated` writes broader than its policy allows, suite 01 test 21 catches it via the structural audit. No proactive sweep deferred — the audit is the standing safeguard.

## Phase 3 — Magic-link auth for the partners app (merged 2026-05-13)

### What landed

- `packages/db`: cookie-aware `createServerClient()` (Server Components / Route Handlers / Server Actions) and `createBrowserClient()` (Client Components) wrappers over `@supabase/ssr`; a Next-15-edge-compatible `createMiddlewareClient()` separate from the server one (different runtime, different cookie adapter); `getMembershipSet()` returning the three sets of memberships (hotels, businesses, strictons-staff slot reserved) the middleware and `/no-access` need; a 7-day `SESSION_MAX_AGE_SECONDS` constant honoured by every cookie write; `auth-types.ts` carrying the `PartnerRole` discriminated union and `MembershipSet` shape.
- `packages/db` migration 14 (`20260505100000_partner_invite_tracking.sql`): `invited_by` / `revoked_at` / `revoked_by` columns on `hotel_memberships` and `business_memberships`; revoke gate policies (revoked rows are filterable but immutable); explicit per-column `INSERT` GRANTs to `authenticated` for invite payloads; the over-broad `update_self_accept` policy dropped entirely (replaced by service-role reconcile in `/auth/confirm`); 14 new pgTAP tests in `tests/11_invite_flow.spec.sql` covering every revoke gate.
- `packages/email`: scaffold (transports + render layer + send wrappers); three transports (`sendgrid`, `console`, `memory`) selected via `EMAIL_TRANSPORT`; hand-rolled inline-styled HTML magic-link template + plain-text fallback; `sendMagicLink()` wrapper that bubbles `EmailSendError` unchanged; the `memory` transport's inbox now lives on `globalThis` (see gotcha) so writer and reader chunks share state in production builds; `MAGIC_LINK_EXPIRY_MINUTES = 15` constant.
- `packages/types`: zod schemas for sign-in (`SignInInputSchema` with open-redirect protection on `next`), invite payloads (`InviteHotelMemberInputSchema`, `InviteBusinessMemberInputSchema`), and revoke payloads (`RevokeMemberInputSchema` with a `scope` discriminator).
- `apps/partners`: `/sign-in` (Server Action submission, Client form for live validation, `/sign-in/check-inbox` confirmation page); `/auth/confirm` Route Handler that consumes the magic link via `verifyOtp`, reconciles invites via service-role, and audit-logs success/failure; middleware that runs `getUser()` + `getMembershipSet()` and routes via the testable pure `decideAuth()` function; `/no-access` landing for the wedged-but-authenticated case (force-dynamic so the rendered email reflects the live session); `/sign-out` POST 303 Route Handler; admin `/members` surface with invite + revoke actions for the calling user's owned hotels and businesses; `lib/audit.ts` service-role `writeAuditLog()` helper; `lib/auth-link.ts` (`buildConfirmUrl`, `resolvePartnersUrl`, `isSafeNextPath`).
- `apps/partners` Sentry: `@sentry/nextjs` ^10 with the v10 hybrid layout (`instrumentation-client.ts` for the browser per Next 15's convention, `sentry.{server,edge}.config.ts` for the runtimes, `instrumentation.ts` registering the `register()` hook + `onRequestError`); `next.config.mjs` wrapped with `withSentryConfig`; SDK + DSN only (no source-map upload, no auth token, no release tracking); `NEXT_PUBLIC_SENTRY_DSN` used uniformly across runtimes so the client bundle inlines a real value.
- `apps/partners` Playwright E2E: webServer pattern boots `pnpm start` against the production build with `EMAIL_TRANSPORT=memory` + `E2E_MODE=1`; one spec covers the request → consume → /no-access → sign-out path end-to-end; `/api/test/last-email` test-only inbox reader gated by `E2E_MODE=1`; `.github/workflows/e2e.yml` boots local Supabase via `supabase start`, extracts `API_URL` / `ANON_KEY` / `SERVICE_ROLE_KEY` from `supabase status -o env`, builds, then runs Playwright.

### Locked decisions

- **`@supabase/ssr` ^0.10, `zod` ^4, `@sentry/nextjs` ^10 — baselined at current majors during commit 1's version check.** Each was past the originally-planned range; with no existing callers the cleanup-cost was zero, so we baselined rather than pin to historical versions.
- **15-minute magic-link expiry, 7-day cookie session.** Magic-link expiry is enforced by Supabase (`MAGIC_LINK_EXPIRY_MINUTES` is a UI display constant, not authority); session expiry is enforced by `SESSION_MAX_AGE_SECONDS` on every `createServerClient()` cookie write. No "remember me" toggle — the 7-day session is the only mode.
- **Supabase's default magic-link email template is disabled.** The dashboard template body is blanked; SendGrid is the sole sender via `welcome@strictons.com`, going through `sendMagicLink()` in `packages/email`. Calling `admin.generateLink({ type: 'magiclink' })` returns the token + hashed-token shape without sending an email; we send our branded version ourselves.
- **Invite reconciliation runs in `/auth/confirm` using the service-role client.** RLS would block a freshly-authenticated user from lifting `user_id` out of `null` on their own membership row (no SELECT-then-UPDATE policy could express it without a self-referential check). The service-role reconcile runs after `verifyOtp` succeeds and writes an audit row regardless of outcome.
- **Magic-link send happens unconditionally on `/sign-in` submission, regardless of whether the email has a membership.** Gating at `/no-access` instead of at submission prevents user-enumeration via response-time differences. The cost is one wasted email per non-member request; the security floor is no leakage of "is this email known to us?".
- **`@sentry/nextjs` SDK + DSN only this phase, no source-map upload.** `SENTRY_AUTH_TOKEN` / `SENTRY_ORG` / `SENTRY_PROJECT` slots intentionally unset; release fingerprinting alone is enough for current volume. `NEXT_PUBLIC_SENTRY_DSN` is the env var name across all runtimes (browser-bundled code requires the prefix; using one name everywhere prevents future drift).
- **Test-data seed is manual in `strictons-dev`.** `dev-test@strictons.com` was inserted as the admin of "Test Beachcomber Hotel" via SQL Editor for verification this phase. Cleanup deferred to Phase 4's admin-app provisioning runbook.
- **`webServer.stdout` / `stderr` set to `'pipe'` in `playwright.config.ts`.** Default is to suppress; piping makes Server Action / Route Handler logs reach the CI output when a test fails. Cost is a few extra lines on failure runs; payoff is not having to re-instrument every time a Playwright failure needs diagnosis.

### Gotchas

- **Migration 14 needed manual application to `strictons-dev`.** `db-deploy-dev.yml` only fires on push-to-main; a long-running feature branch with schema changes drifts dev's schema vs the branch app. The middleware regression at commit 11 was the visible symptom; the actual fault was the migration not having been applied. Fix: ran the migration via SQL Editor + `notify pgrst, 'reload schema'`. Standing rule: when a branch migration changes columns the branch app reads, apply manually to dev or revert until merge.
- **Diagnose before patching — every time.** Repeatedly paid off this phase: middleware regression (was missing migration, not a code bug); Sentry server-side capture silence (env var inlining, not SDK config); `/api/_test` 404s (private-folder routing, not transport state); memory-inbox empty at read (module-instance split, not push-vs-read race). Each one would have been mis-fixed without a ground-truth round.
- **C5 ground-truth dump before writing migration 14.** The `pg_policies` + `column_privileges` + function-body dump surfaced that the planned column-GRANT extension had no existing column GRANT to extend, that `update_self_accept` was over-broad enough that dropping it was cleaner than tightening, and that the `INSERT` GRANT to `authenticated` was net-new ground rather than a reduction. The migration we wrote off the dump was different from (and smaller than) the migration we planned in the abstract.
- **C1 verification of `admin.generateLink` paid off.** Empirically captured the response shape (`properties.hashed_token` paired with `verifyOtp({ type: 'email', token_hash })`) before commit 8 onward depended on it. The masked-response log lived in the repo for one commit and was removed once the shape was locked in.
- **`'use server'` files must export only async functions.** Non-function exports (constants, schemas, type-erased re-exports of values) cause a runtime crash on first import, not a build failure. The `INITIAL_STATE` re-export in `sign-in/actions.ts` and similar shapes in `members/actions.ts` had to move to sibling non-`'use server'` files. The error message — "A 'use server' file can only export async functions" — only appears at request time.
- **The middleware matcher must explicitly exclude `/sign-out`.** Otherwise a middleware-induced 307 on a POST silently re-POSTs to `/no-access` and the sign-out Route Handler never runs. Pre-empted at commit 9 in a JSDoc comment but slipped through to a manual finding at commit 11; the matcher now excludes `_next/static`, `_next/image`, `favicon.ico`, `sign-in`, `sign-out`, `auth/confirm`, `no-access`, `api/test`.
- **pnpm peer-dependency resolution can split a transitive dep when a sibling peer arrives.** Adding `@sentry/nextjs` introduced `@opentelemetry/api` as a peer; pnpm picked a different `next` instance for `apps/partners` than for `packages/db`, breaking type identity across the boundary. Fix: align peer sets explicitly across workspace packages (added `@opentelemetry/api` to `packages/db`'s devDeps; same pattern again later for `@playwright/test`).
- **`@react-email/components`@1.0.12 is npm-deprecated with no replacement.** Hand-rolled the magic-link template HTML (inline styles, table-based layout) rather than ship a deprecated dep. Re-evaluate if a second template arrives — the React-Email-vs-hand-roll calculus changes around N=3.
- **`@sentry/nextjs` v10 uses a hybrid file layout.** `instrumentation-client.ts` is the Next 15 convention for the browser side; `sentry.server.config.ts` and `sentry.edge.config.ts` are the older convention for the server runtimes; `instrumentation.ts` wires both via the `register()` hook + `onRequestError`. Easy to get wrong following v8 tutorials, which still document the four-`sentry.*.config.ts`-files layout.
- **Non-`NEXT_PUBLIC_` env vars in browser-bundled code are inlined to `undefined` at build time.** The same source line — `process.env.SENTRY_DSN` — looks correct on the server (reads from runtime) and broken on the client (the build inlines `undefined`). Only a runtime check (Sentry's `[Sentry Logger] No DSN provided`, or an empty Issues dashboard) reveals the divergence. Standing rule: any env var read from a module that may end up in the client bundle gets the `NEXT_PUBLIC_` prefix.
- **Next.js App Router excludes underscore-prefixed folders from routing.** `app/api/_test/last-email/route.ts` compiled fine, was never registered as a route, and Playwright's polling loop got Next's default 404 page every iteration. The build emits no warning. Standing rule: avoid leading underscores in `app/` folder names; use runtime gates (`E2E_MODE=1`) for test-only routes.
- **Server Actions and Route Handlers load shared package modules as separate instances in Next.js production builds.** A `const inbox: Entry[] = []` at module scope in `packages/email/src/transports/memory.ts` becomes two disjoint arrays — one in the Server Action chunk, one in the Route Handler chunk. Webpack chunk boundary, not a bug. The `[diag][memory]` round at commit `ddd40a5` confirmed it (writer logged `mem-jtf321pl length_after=1`, reader logged `mem-xl2vrd4k length=0`). Fix: store on `globalThis` keyed by `Symbol.for(...)` so both module instances reach the same underlying object via the global symbol registry.
- **Pre-empts in commit messages don't get fixed automatically.** The sign-out matcher exclusion and the diagnostic-vs-patch discipline were both flagged in plan-time JSDoc comments and slipped through to manual findings later. Standing rule: a plan-time pre-empt is either fixed in the same commit or tagged `[FOLLOWUP]` and tracked explicitly. JSDoc-only is not enough.
- **Vercel preview deploys are noticeably slower than production runs.** Cold-start latency dominates middleware (3 Supabase round-trips per protected request: `getUser()`, hotel memberships, business memberships). Production behaves better; the round-trip count is the security floor and not worth optimising at the cost of correctness.
- **GitHub Actions runner backlog is real.** Multiple 14+ minute queue waits during high-load periods this phase. Not a code issue; wait it out, don't retry. Reruns waste a slot and don't move you forward in the queue.
- **Supabase Auth → Sessions controls are Pro-plan-gated.** Time-box, inactivity timeout, and refresh-token reuse settings are paywalled. Free-tier projects enforce session limits via cookie `maxAge` on `createServerClient()` instead — which is what `SESSION_MAX_AGE_SECONDS` is for. Don't burn a session trying to find the dashboard equivalent on a non-Pro project.

### What's deferred

- **Server-side Sentry capture (Node + edge runtimes).** Browser-side capture is verified working; the server-side temporary trigger emitted no Issues despite `NEXT_PUBLIC_SENTRY_DSN` being set on every runtime. Likely the v10 hybrid-layout configuration needs an additional knob; defer to Phase 4 follow-up rather than dig further with no production volume to instrument yet.
- **Source-map upload to Sentry** (`SENTRY_AUTH_TOKEN`, `SENTRY_ORG`, `SENTRY_PROJECT`). Release-tracking precision isn't worth the build-pipeline complexity at current volume.
- **`briefs.data` zod schemas in `packages/types`.** Planned for Phase 3 and cleanly deferred to Phase 5 when the briefs UI lands — the schemas are written once and consumed many times; writing them in isolation with no caller invites premature shape lock-in.
- **Promote/demote member action (admin↔user).** First-invitee auto-promotion via the trigger covers the initial admin; further role changes go through a Strictons-side runbook (service-role) for now. Self-service promote/demote arrives when the partner-app surface needs it.
- **Admin app (`apps/admin`) sign-in.** Phase 4's first commit extends the same auth abstractions to the admin audience.
- **`isStrictonsStaff` slot in `MembershipSet`.** The shape is designed in (the union and the consumers all reference it); the value is hard-coded `false` this phase. Phase 4 wires the actual `strictons_staff` query.
- **Custom hotel domain self-service UI.** Architecture supports it from day one (`hotels.custom_domain` nullable, hostname-or-subpath middleware planned for `mystay`); UI deferred per brief §3.1.
- **Test-data cleanup.** `dev-test@strictons.com` + "Test Beachcomber Hotel" remain in `strictons-dev` for verification continuity; cleanup folds into the Phase 4 admin-app provisioning runbook.
- **E2E coverage beyond sign-in.** Invite/revoke flows have unit-test coverage in `members/actions.test.ts`; Playwright E2E for them is deferred and arrives alongside admin-app E2E in Phase 4.
- **Vercel Pro features.** Upgraded mid-phase for runner priority and dashboard reasons; future phases benefit from priority builds without further action.
- **Latency optimisation.** Production page loads feel slow at low traffic due to Vercel function cold-starts (500–1500ms per cold invocation), three Supabase round-trips per protected request (`auth.getUser` + `hotel_users` + `business_users`), and the US-East default region while users are in AU. Phase 3 ships with no perf work. Anchors for future optimisation: (a) add `syd1` to Vercel function regions for AU / Supabase proximity, (b) `React.cache()` of session data within request lifetime where not already done, (c) potential edge-runtime middleware shift (Sentry edge config already wired). Defer until measured real-user friction or page-load > 2s at warm functions. The one `auth.getUser()` round-trip per request is the security boundary we deliberately don't optimise below.

### Gotchas

- **Sentry server-side silence was the flush-timing issue, not config.** Phase 3's deferred capture gap turned out to be: serverless function freeze drops in-flight Sentry events before the SDK's transport batches them. `@sentry/nextjs` v10's `withServerActionInstrumentation` and `wrapRouteHandlerWithSentry` solve this — both call `vercelWaitUntil(flushSafelyWithTimeout())` in `finally` blocks. Never hand-roll a Sentry wrapper when SDK-native primitives exist; they also cover original-error preservation (next bullet) for free.
- **Next.js production-mode error message sanitisation strips Server Action throws before `onRequestError` sees them.** A bare `throw new Error('specific message')` from a Server Action reaches Sentry as the generic "An error occurred in the Server Components render…" with no diagnostic detail. `withServerActionInstrumentation` calls `captureException(error, { mechanism: ..., type: 'auto.function.nextjs.server_action' })` before Next sanitises — preserving the original message. Same for `wrapRouteHandlerWithSentry` on Route Handlers.
- **`formData` passed to `withServerActionInstrumentation` attaches every form field as Sentry event extras, bypassing `sendDefaultPii: false`.** The wrapper's option attaches each field unconditionally. Default to NOT passing `formData` unless the form is known PII-free. Sign-in (email), invite (invitee email), hotel CRUD (contact_email) all omit `formData` for this reason.
- **`audit_log` append-only trigger interacts with `ON DELETE SET NULL` FKs to fail any cascade attempting to NULL an audit-log column.** Eleven of twelve FKs from `public` tables into `public.users.id` are SET NULL; only `strictons_staff.user_id` is CASCADE. Any user who has signed in has audit history and cannot be deleted. Same applies to hotels, businesses, candidate_businesses. The runbook test-data cleanup section had to adapt twice during verification — final shape is "deactivate, don't delete" for every audited row type.
- **Server Action mutation needs revalidatePath, but revalidatePath alone doesn't fix all click-vs-reload races.** The action's response includes the updated RSC, but `useActionState` state and the parent's row data update on different React paths. The deterministic signal is `state.ok` rendered as visible text; tests/UX must wait on it before triggering a reload. Without that gate, the test races: reload fires before the response is processed; the reload's server-side render's SELECT can read pre-UPDATE state under READ COMMITTED isolation; by the time any diagnostic queries the DB later, the UPDATE has committed and the bug looks mysterious.
- **The "Browser Control button not clickable" report was a useActionState UX issue, not a wiring problem.** Server-side capture was working all along during the Phase 4 commit 2 diagnostic; client-side simply didn't re-render to confirm the click registered. Generalisation: cross-check UI reports against the actual server-side signal (Vercel logs, Sentry events, DB state) before assuming the UI surface is the bug.
- **Vercel's "Redeploy" button defaults to rebuilding the same commit, not the latest.** The dialog has a "use latest commit on the branch" checkbox that must be explicitly chosen. A redeploy without this option rebuilds the old code under a new URL — invisible footgun.
- **`@sentry/nextjs` v10 has SDK-native wrappers for both Server Actions and Route Handlers.** `withServerActionInstrumentation(name, options?, callback)` for the former; `wrapRouteHandlerWithSentry(handler, { method, parameterizedRoute })` for the latter. Same shape, same benefits. No reason to hand-roll either pattern.
- **Vitest mocks for Next-runtime helpers.** `revalidatePath` from `next/cache` throws "Invariant: static generation store missing" when called outside a Next request context. Unit tests for Server Actions that call `revalidatePath` must mock it. The mock pattern is in `apps/partners/app/(protected)/members/actions.test.ts` and `apps/admin/app/(protected)/hotels/actions.test.ts`; reuse for Phase 5+ Server Action tests.
- **`'use server'` files reject non-async-function exports at runtime, not build time.** Phase 3 paid for this once; Phase 4 pre-empted it across every new action module. Constants and type definitions live in sibling files (`types.ts` next to `actions.ts`). The check fires on first import per request, so the failure surfaces as a 500 at the first form submission, not at build.
- **TypeScript exhaustiveness for cross-package enums.** `@strictons/types` doesn't depend on `@strictons/db`; the DB enum literal array is restated in the types package and the exhaustiveness check (`Exclude<DbEnum, MyEnum> extends never`) lives in the consumer (admin's `actions.ts`) where `@strictons/db/types` is already a dep. Keeps the types package's runtime dep graph light for marketing / mystay. If a future enum becomes shared between three+ apps, promote the check into a shared `@strictons/db/enum-checks` subpath.

Surfaced in commit 5's report ("admin-context tweak"), challenged on the review, reverted in a separate fix-up commit (not amended into commit 5, since the expansion originated in commit 4 and amending would have left commit 4's surface untouched). **Lesson reinforced as a standing pre-empt: when an approved contract is being extended in any way — return-shape additions, new union variants, new fields — pause and surface the extension as a question BEFORE writing it, not after. The bar isn't "is the extension defensible"; the bar is "was this in the approved contract."**
- **Admin app had zero Playwright infrastructure before Phase 5.** Commit 6's prompt was originally scoped as "write the spec," but the first action surfaced that the admin app had no `playwright.config.ts`, no `e2e/` dir, no `test:e2e` script, no `/api/test/last-email` route, and `@playwright/test` wasn't a direct dep. Commit 6 expanded to "scaffold admin-side Playwright + write the spec." Right escalation; the scaffolding lives in the right places (mirroring partners-side patterns byte-for-byte where applicable, especially the E2E_MODE gate). Generalisation for future phases: any "write the test" prompt against an app that hasn't had an E2E pass before will probably surface this same scaffolding gap; budget for it.
- **The `redirectTo` URL construction is single-segment `` `${partnersUrl}/auth/confirm?next=${encodeURIComponent(next)}` `` matching the partners-side action exactly.** A commit summary briefly described it with nested backticks that rendered as a doubly-nested URL in chat output — a markdown artifact, not a code issue. Verified by reading the file verbatim against the partners-side reference. The right escalation pattern when challenged ("quote the file verbatim") is the durable way to disambiguate "summary mangling vs real bug."
- **Vercel-preview-vs-CI-local-Supabase isolation incompatibility.** The original commit-1 framing of `E2E_PARTNERS_URL` as "construct the Vercel preview URL per branch" did not account for the existing `e2e.yml` booting a fresh local Supabase per CI run. A test invitee in the local Supabase doesn't exist in `strictons-dev` (where Vercel previews point), so any cross-app spec hitting a Vercel preview would fail for infrastructure reasons unrelated to the code. Caught at the plan-pause in commit 6 by reading `e2e.yml` before mirroring the assumed pattern — the right discipline before any "cross-environment" assumption.
- **`pnpm format:check` is a distinct CI gate from `pnpm typecheck` and `pnpm lint`.** Phase 5 commit 6 introduced two files with prettier formatting drift (one from commit 3, one from commit 5); local pre-push ran typecheck + lint but not format:check. CI caught it; one-line fix-up resolved. Standing pre-empt going forward (see locked decisions).
- **CI was the first real signal for the cross-app spec.** Local execution wasn't possible because the development sandbox has docker CLI but no daemon, so the local Supabase boot via `supabase start` can't run. This is an environmental limit, not a process failure. Future Phase 6+ work that adds new CI infrastructure (workflow files, sandboxed runners, env-var matrices) should expect this same "static verification only, CI is the first real run" pattern. The fix when CI is the first signal is to be tight on static gates locally (`typecheck` + `lint` + `format:check` + unit tests + `playwright test --list`) and prepared for a CI-driven diagnostic round if needed.
- **`vitest` doesn't read TypeScript path aliases without a `vitest.config.ts`.** The admin app had `paths: { "@/*": ["./*"] }` in `tsconfig.json` from Phase 4 but no vitest config — Phase 4's tests didn't import via the alias so it didn't matter. The Phase 5 `requireStaff` lift introduced the first `@/lib/...` import inside an admin test, and the existing test suite went red on the first re-run. Added `apps/admin/vitest.config.ts` mirroring `apps/partners/vitest.config.ts`. Resolution took seconds once the symptom was identified; the lesson is "any new path-alias use in a test file might require a vitest config that the production runtime doesn't need."
